# Prompt 13: Tkinter GUI

Date: 2026-06-20

## GUI Entry Point And Behavior Summary

- Added `src/dwca_cloud_geospatial/gui.py`.
- Added console script `dwca-cloud-geospatial-gui`.
- Added module entry point `python -m dwca_cloud_geospatial.gui`.
- The GUI lets users choose an input DwC-A archive or unpacked archive
  directory and an output bundle directory.
- FlatGeobuf is selected by default; GeoParquet is explicit. Both formats can
  be selected together.
- Existing output paths are guarded by an overwrite checkbox before conversion
  starts.
- Optional validation after conversion is enabled by default.
- GeoParquet large-output mode is exposed only as a GeoParquet option and maps
  to implemented `GeoParquetWriterOptions(large_output_mode=True)` behavior
  plus conversion chunk size. Partitioned GeoParquet output is not exposed.
- The GUI status panel shows accepted/rejected counts, manifest and viewer
  paths, generated geospatial artifact paths, retained GeoPackage staging
  artifact paths when present, non-fatal warnings and validation summaries.
- Validation display separates required validation errors, validation warnings
  and dependency-dependent skipped checks.
- Viewer guidance points at the copied `<output>/index.html` entry, explains
  that FlatGeobuf bundles use `data/occurrences.fgb` as the MVP browser map
  layer, treats `data/occurrences.gpkg` as artifact/download metadata only,
  and describes GeoParquet-only bundles as valid no-map-layer
  metadata/provenance states.

The GUI does not create standalone packaged desktop binaries and does not
start a backend service.

## Core API Reuse Evidence

- GUI conversion options are built by `build_conversion_options()` and passed
  to `dwca_cloud_geospatial.conversion.convert_dwca_archive`.
- GUI validation calls
  `dwca_cloud_geospatial.validation.validate_output_bundle`.
- GUI code does not parse DwC-A rows, normalize records, write FlatGeobuf,
  GeoParquet, GeoPackage, manifests, processing metadata or rejected-record
  reports directly.
- Conversion failures preserve core `ConversionError.message` and parser
  diagnostics.
- Non-fatal warnings are read from the core conversion result:
  normalization warnings from `normalization_result.warnings` and FlatGeobuf
  writer warnings from `flatgeobuf_result.warnings`.

## Follow-Up Copy Behavior

After manual GUI use, viewer instructions were visible but hard to copy from
the read-only status text area. Added explicit copy behavior:

- `Ctrl+C` and `Cmd+C` copy the selected status text, or the full status text
  when nothing is selected.
- Right click, middle click and control-click show a context menu with
  `Copy` and `Copy all`.
- Added a `Copy Text` button that copies the full current status panel,
  including viewer instructions.

Verification:

```bash
.venv/bin/python -m compileall -q src tests
.venv/bin/python -m pytest tests/test_gui.py -q
git diff --check
```

Result: compile and diff checks passed; GUI-adjacent tests reported
`8 passed`.

Follow-up: context-menu copy worked in manual GUI use, but `Ctrl+C`/`Cmd+C`
did not reliably copy selected status text because keyboard focus could remain
outside the read-only `Text` widget after mouse selection. Added explicit focus
handling for the status text area and app-level copy bindings for `<<Copy>>`,
`Ctrl+C` and `Cmd+C`. The app-level handler only intercepts copy when the
status text has focus or owns a selection, so normal input-field copy behavior
is preserved.

Second follow-up: `Ctrl+C`/`Cmd+C` still did not work reliably on the local
Tk/macOS path while the status text used `state="disabled"`. The status panel
now stays in normal `Text` state so native copy bindings can run, while
editing keys are blocked separately to keep it effectively read-only.

Third follow-up: `Ctrl+C`/`Cmd+C` still did not work in manual GUI use. The
remaining issue was likely the broad read-only `<Key>` blocker catching copy
keystrokes with platform-specific modifier state before the copy binding could
complete. Removed the broad key blocker. The status text now relies on native
`Text` copy behavior and only blocks explicit mutation actions such as cut,
paste, backspace and delete.

Fourth follow-up: `Ctrl+C`/`Cmd+C` remained unreliable in manual GUI use.
Accepted this as an MVP limitation because context-menu copy and the `Copy
Text` button are sufficient for now. Updated the static viewer no-map-layer
message so GeoParquet-only bundles explicitly tell users to generate with
FlatGeobuf selected if they want occurrence points to appear on the map.

Fifth follow-up: audited downstream prompts `14_demo_docs_hardening.md` and
`15_gbif_doi_citation.md` for Prompt 13/current-dialog handoff coverage.
Updated Prompt 14 with GUI launch troubleshooting, copy behavior/limitations
and the new viewer no-FlatGeobuf message. Updated Prompt 15 because it edits
viewer provenance and may touch conversion options: it now explicitly
preserves the no-FlatGeobuf message, existing GUI behavior and GUI/docs update
checks when conversion options change.

Sixth follow-up: audited project documentation for Prompt 13/current-dialog
handoff coverage. Updated `docs/converter.md`, `docs/developer_setup.md`,
`docs/development_plan.md`, `docs/viewer_contract.md` and `viewer/README.md`
so the docs preserve GUI launch troubleshooting, context-menu/`Copy Text`
copy behavior, the `Ctrl+C`/`Cmd+C` MVP limitation and the updated
no-FlatGeobuf viewer message.
